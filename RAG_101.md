## 1. The Problem RAG Solves

Large Language Models (LLMs) like GPT-4 are trained on enormous amounts of text
from the internet. That training process encodes knowledge into the model's
billions of parameters — its "memory."

But that memory has three serious problems:

### Problem 1 — Knowledge Cutoff

Training is expensive and slow. A model trained in early 2024 knows nothing about
events in late 2024 or 2025. If you ask it about recent news, it either makes
something up or tells you it does not know.

### Problem 2 — No Private Knowledge

The model was trained on public internet data. It knows nothing about:
- Your company's internal documents
- A PDF you uploaded
- Your codebase
- A customer's support history

### Problem 3 — Hallucination

When a model does not know something, it often invents a confident-sounding
answer anyway. This is called **hallucination**. The model is not lying — it is
doing what it was trained to do (predict plausible text) — but the result is
dangerously wrong information presented with full confidence.

### The Old Solutions (and why they fall short)

| Solution | What it does | Problem |
|----------|-------------|---------|
| Prompt stuffing | Paste the whole document into the prompt | Context window limits, expensive per call |
| Fine-tuning | Retrain the model on new data | Costs thousands of dollars, takes weeks, knowledge still goes stale |
| Just accept it | Hope the model knows | Hallucination, outdated info |

**RAG solves all three** — cheaply, in real time, without retraining the model.

## 2. What RAG Actually Is

**Retrieval-Augmented Generation (RAG)** was introduced by researchers at Meta AI
in a 2020 paper. The core idea is simple:

> Before the LLM generates an answer, *retrieve* the most relevant pieces of
> information from a knowledge base and *include them in the prompt* as context.

The model never guesses from memory. It reads the relevant passages and answers
based on what it just read — exactly like a human looking something up before
answering a question.

### The Three-Word Version

    Retrieve  ->  Augment  ->  Generate

- **Retrieve:** Find the most relevant document chunks for the user's question
- **Augment:** Add those chunks to the prompt as context
- **Generate:** Let the LLM produce an answer grounded in that context

## 3. The Full Architecture

There are two separate phases:

    INDEXING PHASE  (done once, ahead of time)
    ============================================
    Raw Documents
        |
        v  Document Loader
    list of Document objects
        |
        v  Text Splitter
    list of Chunks   <-- small overlapping passages
        |
        v  Embedding Model
    list of Vectors  <-- each chunk as numbers
        |
        v  Vector Database
    Indexed Vector Store  <-- searchable by similarity


    QUERY PHASE  (happens every time a user asks a question)
    ============================================
    User Question
        |
        +------------------------------------------+
        |                                          |
        v  Embedding Model                         |
    Question Vector                                |
        |                                          |
        v  Similarity Search                       |
    Top-K Relevant Chunks                          |
        |                                          |
        v  format_docs()                           |
    Context String                                 |
        |                                          |
        +---------- Prompt Template <--------------+
                          |        (original question)
                          v
                   LLM (gpt-4o-mini)
                          |
                          v
                    Final Answer

## 4. Component 1 — Documents & Loaders

### What is a Document?

In LangChain, a Document is a simple Python object with two fields:

    document.page_content  -- the actual text (string)
    document.metadata      -- dict with source info
                              e.g. {"source": "report.pdf", "page": 3}

### What is a Loader?

A loader knows how to read a specific data source and convert it into a list of
Document objects. LangChain has loaders for almost everything:

| Source | Loader class |
|--------|-------------|
| PDF file | PyPDFLoader |
| Folder of files | DirectoryLoader |
| Website | WebBaseLoader |
| Plain text | TextLoader |
| YouTube transcript | YoutubeLoader |
| Notion database | NotionDBLoader |
| Wikipedia | WikipediaLoader |
| CSV file | CSVLoader |

Every loader has the same interface:

    loader = PyPDFLoader("my_report.pdf")
    documents = loader.load()
    # returns [Document(page_content="...", metadata={"page": 0}), ...]

## 5. Component 2 — Chunking

### Why Not Just Index the Whole Document?

**Embedding quality degrades with length.**
An embedding converts text to a fixed-size vector. A 50,000-character document
embedded as one vector produces a blurry average of every topic in it. When you
search for "quarterly revenue" you might surface a chunk that vaguely covers
everything instead of the specific table you need.

**Context window limits.**
Even if retrieval was perfect, you cannot paste 10 entire documents into a
prompt. You need to be selective.

### How Chunking Works

You split each Document into smaller, overlapping pieces:

    Original: [================================================]

    Chunk 1:  [======================]
    Chunk 2:               [======================]
    Chunk 3:                            [======================]
                            <-- overlap -->

The overlap (typically 10-20% of chunk size) ensures a sentence split across two
chunks appears complete in at least one of them.

### RecursiveCharacterTextSplitter

The most common splitter. It tries natural boundaries in this order:
1. Paragraph breaks  (newline newline)
2. Line breaks       (newline)
3. Sentence ends     (". ")
4. Word boundaries   (" ")
5. Characters        (last resort)

Chunks almost always end at a natural boundary rather than mid-word.

### Choosing Chunk Size

| Use case | Chunk size | Why |
|----------|-----------|-----|
| Factual Q&A | 256-512 chars | Precise, specific retrieval |
| Summarization | 1024-2048 chars | More context per chunk |
| Code | 512 chars | Function/class level |
| Legal documents | 512-1024 chars | Clause-level precision |

There is no perfect answer. Run evaluation (Section 12) to find the best value
for your specific documents and questions.

## 6. Component 3 — Embeddings

### What Is an Embedding?

An embedding model is a neural network that converts any string of text into a
fixed-length list of numbers — a **vector**. The key property:

> Texts with similar meaning produce vectors that are close together.
> Texts with different meanings produce vectors far apart.

    "What is RAG?"      ->  [0.021, -0.134, 0.872, ..., 0.003]   1536 numbers
    "RAG overview"      ->  [0.019, -0.130, 0.869, ..., 0.001]   very close!
    "Recipe for pasta"  ->  [-0.412, 0.823, -0.011, ..., 0.234]  very different

This is how retrieval works: convert the question to a vector, find the chunk
vectors nearest to it, return those chunks as the relevant context.

### How Closeness Is Measured — Cosine Similarity

The standard measure is cosine similarity. Imagine each vector as an arrow
pointing in a direction in space. Cosine similarity measures the angle between
two arrows:

    similarity = 1.0   same direction  (very similar meaning)
    similarity = 0.0   perpendicular   (unrelated)
    similarity = -1.0  opposite        (opposite meaning)

### Common Embedding Models

| Model | Dimensions | Cost | Notes |
|-------|-----------|------|-------|
| text-embedding-3-small (OpenAI) | 1536 | ~$0.02/1M tokens | Fast, strong baseline |
| text-embedding-3-large (OpenAI) | 3072 | ~$0.13/1M tokens | Higher accuracy |
| all-MiniLM-L6-v2 (HuggingFace) | 384 | Free, local | Good for most tasks |
| embed-english-v3.0 (Cohere) | 1024 | Paid API | Strong multilingual |

### Critical Rule: Consistency

You MUST use the same embedding model at indexing time and at query time.
If you embed chunks with text-embedding-3-small, you must also embed the query
with text-embedding-3-small. Mixing models produces garbage results because
the two vector spaces have completely different geometry.

## 7. Component 4 — Vector Database

### What It Stores

A vector database stores your chunk vectors alongside the original text:

    id      : "chunk_042"
    vector  : [0.021, -0.134, 0.872, ...]   1536 numbers
    text    : "RAG was introduced by Lewis et al. at Meta AI in 2020..."
    metadata: {"source": "wikipedia", "chunk_index": 42}

### Why Not a Regular Database?

A SQL query like WHERE text LIKE '%RAG%' does keyword matching — it only
finds exact words. A vector search finds semantically similar content even
when the words are completely different.

Example:
    Query: "How does retrieval-augmented generation reduce errors?"
    Chunk: "RAG grounds responses in retrieved evidence, reducing hallucination"

    Keyword search: NO MATCH (different words)
    Vector search:  HIGH SIMILARITY (same concept)

### Common Vector Databases

| Database | Type | Free | Best for |
|----------|------|------|---------|
| ChromaDB | Local / embedded | Yes | Development, demos |
| Qdrant | Cloud or self-hosted | Yes (cloud) | Production |
| Pinecone | Cloud only | Yes (limited) | Managed production |
| FAISS | Local library | Yes | Research, offline |
| pgvector | PostgreSQL extension | Yes | If you already use Postgres |

## 8. Component 5 — Retriever

### What a Retriever Does

A retriever wraps your vector store and provides one method: invoke(question).
It embeds the question, searches the store, and returns the top-K most similar
chunks as Document objects.

    retriever = vector_store.as_retriever(
        search_type="similarity",
        search_kwargs={"k": 4}
    )
    results = retriever.invoke("What problem does RAG solve?")
    # returns [Document, Document, Document, Document]

### Search Types

| search_type | How it works | Use when |
|-------------|-------------|---------|
| "similarity" | Return k nearest vectors | Default — works for most cases |
| "mmr" | Balance relevance AND diversity | Top-k chunks are repetitive |
| "similarity_score_threshold" | Only return above a score cutoff | Avoid irrelevant results |

### Choosing K

K = how many chunks to return per query.

    Too low  (k=1-2): Fast, cheap, may miss important context
    Too high (k=8+):  More context, but wastes tokens, can confuse the LLM
    Sweet spot: k=3-5 for most use cases

## 9. Component 6 — Prompt Template

### What It Does

The prompt template combines the retrieved chunks (context) and the user's
question into a single structured message that the LLM receives.

### Why You Need a System Prompt

Without instructions, the LLM will blend its training knowledge with the context,
which defeats the purpose of RAG. Your system prompt must:

1. Constrain the model: "Answer ONLY using the context below"
2. Provide a fallback: "If the answer is not in the context, say I don't know"
3. Include a {context} placeholder where the retrieved chunks go

### Example Template

    SYSTEM:
    You are a helpful assistant. Answer the user's question using ONLY
    the context passages below. Do not use outside knowledge.
    If the answer is not in the context, say "I don't know."

    Context:
    {context}

    HUMAN:
    {question}

The {context} slot is filled by your format_docs() function.
The {question} slot is filled by the user's actual query.

## 10. Component 7 — The LLM

### Role in RAG

The LLM is the last step — it reads the formatted prompt (question + retrieved
context) and writes the answer. In RAG, the LLM acts as a **reader and writer**,
not a memory store.

### Why temperature=0?

Temperature controls randomness in the output:

    temperature=0      Deterministic — same input always gives same output
    temperature=0.7    Creative — some variation in each response
    temperature=1.0+   Unpredictable — high randomness

For RAG you almost always want temperature=0 because:
- Reproducibility matters for evaluation and debugging
- You want the model to stick to the facts in the context, not get creative
- Consistent outputs make it easier to compare pipeline versions

### Common LLM Choices for RAG

| Model | Speed | Cost | Quality |
|-------|-------|------|---------|
| gpt-4o-mini | Fast | Low | Good — best value for RAG |
| gpt-4o | Medium | High | Excellent — use for production |
| claude-3-haiku | Fast | Low | Good alternative |
| llama3 (local via Ollama) | Depends on hardware | Free | Good, no API needed |

## 11. Putting It All Together — The Query Flow

Here is exactly what happens every time a user submits a question:

    Step 1: User types "What problem does RAG solve?"

    Step 2: The question is embedded
            "What problem does RAG solve?" -> [0.021, -0.134, ...]

    Step 3: The vector store is searched
            Find the 4 chunks whose vectors are closest to the question vector.

    Step 4: Top-4 chunks are returned
            Chunk A: "LLMs rely on static training data which can become outdated..."
            Chunk B: "RAG was introduced to give models access to external knowledge..."
            Chunk C: "Without retrieval, models often hallucinate facts..."
            Chunk D: "The retriever finds relevant passages at inference time..."

    Step 5: Chunks are formatted into a context string
            format_docs([A, B, C, D]) -> one big string with all 4 passages

    Step 6: Prompt is assembled
            SYSTEM: Answer using only the context below.
                    Context: [the 4 chunks joined together]
            HUMAN:  What problem does RAG solve?

    Step 7: Prompt is sent to the LLM

    Step 8: LLM generates the answer
            "RAG solves three problems with standard LLMs: knowledge cutoff,
             lack of private data access, and hallucination. By retrieving
             relevant passages at query time..."

    Step 9: Answer is returned to the user

## 12. Evaluating a RAG System

### Why Evaluation Matters

Without metrics, you are guessing. You might think your pipeline works well
because a few test answers look good, but there could be systematic failures
you haven't noticed.

### The 4 RAGAS Metrics

RAGAS is the standard evaluation framework for RAG. It uses the LLM itself
as a judge to score four dimensions:

**Faithfulness** (0.0 - 1.0)
    Question: Is every claim in the answer supported by the retrieved context?
    How scored: Breaks answer into atomic claims, checks each against context
    Low score means: The LLM is hallucinating — inventing things not in the context

**Answer Relevancy** (0.0 - 1.0)
    Question: Does the answer actually address what was asked?
    How scored: LLM generates questions that the answer implies, compares to original
    Low score means: Answers are vague, off-topic, or miss the point

**Context Recall** (0.0 - 1.0)
    Question: Does the retrieved context contain the information from the ground truth?
    How scored: Checks what fraction of ground truth sentences appear in the context
    Low score means: The retriever is missing relevant chunks (wrong k, bad chunk size)

**Context Precision** (0.0 - 1.0)
    Question: Are most of the retrieved chunks actually relevant?
    How scored: Checks what fraction of retrieved chunks contributed to the answer
    Low score means: Too much irrelevant context is being retrieved (noisy retriever)

### What Good Scores Look Like

    Score >= 0.85    Excellent
    Score >= 0.70    Good
    Score >= 0.50    Needs work
    Score < 0.50     Serious problem

### The Golden Dataset

To run evaluation you need a golden dataset — a set of questions with known
correct answers that YOU wrote (not generated by the model). This is your
ground truth.

Each row has:
    question      the user's question
    ground_truth  your reference answer (written before running the model)
    answer        the answer your pipeline generated
    contexts      the chunks retrieved for that question (raw text strings)

Aim for at least 10-20 question/answer pairs covering different topics in
your document. Fewer gives statistically unreliable averages.

## 13. RAG vs Fine-Tuning

These are the two main ways to give an LLM new knowledge. They are very different:

### Fine-Tuning

You take a pre-trained model and continue training it on your new data.
The knowledge bakes into the model's weights permanently.

    Pros: Fast inference (no retrieval step), knowledge is part of the model
    Cons: Expensive ($100-$10,000+), takes hours to days, knowledge goes stale,
          requires labeled training data, risk of catastrophic forgetting

### RAG

You keep the model unchanged. At query time, you retrieve relevant passages
and feed them as context.

    Pros: Cheap, knowledge stays fresh (update the vector store anytime),
          explainable (you can see which chunks were used), no retraining
    Cons: Retrieval adds latency, retriever can fail, context window limits

### When to Use Which

| Use RAG when | Use Fine-Tuning when |
|-------------|---------------------|
| Knowledge changes frequently | Knowledge is stable and won't change |
| You need to cite sources | You need to change the model's style or tone |
| You have private documents | You have a huge labeled dataset and budget |
| Budget is limited | Response speed is critical and latency matters |
| You want explainability | The task requires deep domain specialization |

### The Hybrid Approach

Many production systems use both: fine-tune the model on domain vocabulary and
tone, then use RAG for up-to-date factual retrieval.

## 14. Common Failure Modes & Fixes

### Failure 1 — Hallucination (Low Faithfulness)

    Symptom: Answers contain claims not found in any retrieved chunk
    Root cause: System prompt is too weak, or temperature too high
    Fix 1: Strengthen system prompt — "You MUST only use the context below"
    Fix 2: Set temperature=0
    Fix 3: Switch to a stronger LLM (gpt-4o vs gpt-4o-mini)

### Failure 2 — Wrong Chunks Retrieved (Low Context Recall)

    Symptom: The right information exists in the document but is not retrieved
    Root cause: Chunks are too large (blurry embeddings) or k is too small
    Fix 1: Reduce chunk_size (try 256 instead of 512)
    Fix 2: Increase k (try k=6 or k=8)
    Fix 3: Try hybrid search (BM25 keyword + dense vector combined)
    Fix 4: Try MMR retrieval to improve diversity

### Failure 3 — Too Much Noise (Low Context Precision)

    Symptom: Retrieved chunks are mostly irrelevant to the question
    Root cause: k too high, chunk_size too small, or similarity threshold too low
    Fix 1: Reduce k
    Fix 2: Increase similarity score threshold
    Fix 3: Add metadata filtering (only search specific document sections)
    Fix 4: Use a reranker to re-score chunks after initial retrieval

### Failure 4 — Vague Answers (Low Answer Relevancy)

    Symptom: Answers are technically from the context but don't address the question
    Root cause: Weak prompt, or the question is ambiguous
    Fix 1: Improve the system prompt with more explicit instructions
    Fix 2: Add query rewriting (LLM rephrases the question before retrieval)
    Fix 3: Add chain-of-thought reasoning to the prompt

### Failure 5 — Model Says "I Don't Know" Too Often

    Symptom: The answer is in the document but the model refuses
    Root cause: The right chunks are not being retrieved (retrieval failure)
    Fix: Same as Failure 2 above — improve retrieval, not the generation step

## 15. Key Vocabulary Cheat Sheet

| Term | Definition |
|------|-----------|
| **RAG** | Retrieval-Augmented Generation — grounding LLM answers in retrieved documents |
| **LLM** | Large Language Model — e.g. GPT-4, Claude, Llama |
| **Document** | Raw source material (PDF, webpage, text file) |
| **Chunk** | A small passage split from a document, typically 256-1024 characters |
| **Chunk overlap** | Characters re-used between adjacent chunks to prevent boundary cut-offs |
| **Embedding** | A fixed-length vector of numbers representing the meaning of text |
| **Vector** | A list of numbers — the numeric form of an embedded chunk or query |
| **Vector space** | The high-dimensional numeric space where vectors live |
| **Cosine similarity** | Measure of angle between two vectors — 1.0 = identical, 0.0 = unrelated |
| **Vector database** | Database optimised for storing and searching vectors by similarity |
| **ChromaDB** | Local vector database — runs in memory, no cloud account needed |
| **Retriever** | Component that takes a question and returns the most similar chunks |
| **Top-K** | The K chunks with highest similarity to the query |
| **MMR** | Maximal Marginal Relevance — retrieval that balances relevance and diversity |
| **Prompt template** | A reusable message structure with {context} and {question} placeholders |
| **LCEL** | LangChain Expression Language — the pipe (|) syntax for chaining components |
| **Context window** | Maximum tokens an LLM can process in one call |
| **Hallucination** | When an LLM confidently states something false not supported by evidence |
| **Golden dataset** | Hand-written Q&A pairs used as ground truth for evaluation |
| **RAGAS** | Evaluation framework for RAG — scores faithfulness, relevancy, recall, precision |
| **Faithfulness** | Are answer claims grounded in the retrieved context? |
| **Answer relevancy** | Does the answer address what was actually asked? |
| **Context recall** | Does retrieved context cover the ground truth information? |
| **Context precision** | Are most retrieved chunks relevant (low noise)? |
| **Fine-tuning** | Retraining a model's weights on new domain-specific data |
| **Indexing phase** | One-time setup — loading, chunking, embedding, storing |
| **Query phase** | Real-time — embed question, retrieve, prompt, generate |
| **Temperature** | LLM randomness setting — 0 = deterministic, 1+ = creative |
